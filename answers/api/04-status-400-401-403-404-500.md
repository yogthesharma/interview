# When do you return 400 vs 401 vs 403 vs 404 vs 500?

**Target time:** 45 seconds

---

## Talk track

| Code | Meaning | When |
|------|---------|------|
| **400** | Bad Request | Invalid input, failed validation, malformed JSON |
| **401** | Unauthorized | Not authenticated — missing/invalid token |
| **403** | Forbidden | Authenticated but **not allowed** (wrong tenant, role) |
| **404** | Not Found | Resource doesn't exist (or hidden for security) |
| **409** | Conflict | Duplicate, state conflict (already submitted) |
| **422** | Unprocessable | Valid JSON but business rule failed (sometimes) |
| **500** | Internal Server Error | Unexpected bug — log, generic message to client |

> **401 vs 403:** 401 = "who are you?"; 403 = "I know you, you can't do this."  
> **404 vs 403:** sometimes return **404** for cross-tenant access to avoid leaking existence.

---

## Code

```json
// 400
{ "error": "Validation failed", "fieldErrors": { "email": "Invalid format" } }

// 401
{ "error": "Unauthorized" }

// 403
{ "error": "Forbidden", "message": "Insufficient permissions" }
```

---

## Avoid

- 500 for validation errors (client's fault → 400)
