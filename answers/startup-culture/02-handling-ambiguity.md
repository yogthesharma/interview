# How do you handle ambiguity?

**Target time:** 30 seconds

---

## Talk track

> **Clarify the goal, not every step** — what's the user outcome? deadline? constraints?
>
> **Small bet first** — spike or thin slice, show something, get feedback. At Atlys visa flows, requirements shifted with policy changes — I'd ship a workable path and iterate.
>
> **Write it down** — short PRD comment, ticket, or Slack summary so alignment doesn't live only in my head.
>
> **Ask early** — one focused question beats building the wrong thing for a week.
>
> Ambiguity is normal in startups; paralysis is the failure mode I avoid.

---

## Avoid

- "I need perfect specs" — wrong fit for startup
