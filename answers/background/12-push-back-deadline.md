# Tell me about a time you had to push back on a deadline.

**Target time:** 60–90 seconds (STAR format)

---

## Talk track (scope + quality at Atlys — customize if needed)

> **Situation:** At Atlys we had a **visa onboarding feature** tied to a marketing launch. Product wanted everything in one release — new UI steps, API changes, and **performance work** on Core Web Vitals.
>
> **Problem:** The deadline was realistic for "something shipped," but not for **all three without risk** — especially perf on a high-traffic flow where bad LCP hurts conversion.
>
> **What I did:** I pushed back with a **scope split**, not a flat "no":
> - **Phase 1 (deadline):** Core flow + API — shippable, tested, behind feature flag if needed
> - **Phase 2 (one sprint later):** CWV optimizations — lazy loading, code splitting — with metrics to prove impact
>
> I brought **data** — bundle size estimates, which steps were perf-critical — and offered to **own Phase 1** so the launch wasn't blocked.
>
> **Result:** We hit the launch with the core flow. Perf work landed right after with measurable LCP improvement. No firefighting weekend.
>
> **Learning:** Push back with **options**, not excuses. Stakeholders remember solutions.

---

## Bullet spine

1. Launch pressure — real business date
2. Risk if everything at once
3. Phased scope — you proposed the split
4. You owned delivery on phase 1
5. Better outcome than a rushed monolith release
